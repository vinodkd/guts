The Grand Unified Theory of Software

The big idea
Define code as structures a la mechanical primitives
Create rules to represent applications as aggregates of such primitives
model notions of stability, brittleness, etc as functions of the aggregate structure
Model things that cause code to change structure or behavior as forces that act on it and derive ways of predicting their impact
Model the subjective nature of such a theory by including the human observer within it
Model the "Turtles all the way down" nature of software
Try to come up with "natural" formulas, ie those that do not need empirical constants to determine their value

statements are the atoms of code
they have context and a user at their level
meaning at a level above them
implementation at a level below them
so statement can be represented as: s: (M,C,U, B)
M: Meaning - facts/ s^1
meaning is always orthogonal to Base and the set of executors made available
meaning is expressed in an informal way eg, English, to denote that its not directly amenable to mechanical processing
it is possible to ascribe FOPL (or even imperative) semantics to meaning.Some gain in higher-order calculation is obviously possible, but Godel's limits render it eventually pointless.
the point of representing meaning is that processes occuring inside the user's mind can be modeled as being contained in it
C: Context: (I, E, O, S)
I: Input: Inputs to the statement
E: Executor: the part of the base that knows how to execute the statment
O: Output: Outputs from the statement
S: Side effects: Impacts to the context as a result of executing the statment
U: User: who observes, optionally provides input and/or on whose behalf the execution happens
B: Base: (Eb, Sb)
A Base B is a collection of executors Eb that can execute a set of statements Sb

gravity (gB): an arbitrary number that denotes the size of the base

gB= c1 * |Eb|, where c1 is a measure of the overall "icebergness" of the base.

gB should be used as a measure of comparing two bases.
